• Blackjack is a gambling game played with cards on a green table
  • It was my choice for project #1 at GA-DC WDI last summer...
  • Our specs: javascript, object-oriented, DRY, commented...
  • And as always at GA: "Show us what you can do!!!"

You can see the original version repo here, but for the upgraded and refactored version I wanted to try something different. I chose a grid design to explore ways of manipulating the DOM and playing with design alternatives. A quick coding test suggested the basic functionality would be fairly straight-forward:

  • make a 12 by 18 table of cells
  • select them individually or in groups
  • apply classes and other attributes at will
  • append images or divs to create layers or features above the grid
  • store parameters and behaviors in screen, player, display and game objects
Design
 
 
 
 
 
The grid is a 12 x 18 table of 50x50px cells.
Game components are generated by defining regions on the grid.
Regions are located via their indexCell which has row and column attributes, and are further defined by width, height and merge attributes
Width and height define size (in the number of cells wide or tall)
A merged region is created by a single spanned cell; ummerged regions result from a series of individual cells.
 
Rowspans and colspans had to be considered when locating the index cells on affected rows.
In this example, an index cell in column 13 as actually in column 10 due to the spanned cells in the same row, so it's 'cell address' needs to be adjusted programatically.
Tracking rowspans can be more challenging.
When an index cell is in the same row, rowspans have no effect.
If the index cell is in a row containing spanned cells, the number of columns in that row is reduced.
Finding rowspanned cells requires looking at all rows above the target row to check for rowspans -- a cumbersome process.
My solution was to set up a table of rowspaned cell records, each of which is initialize to FALSE.
When a rowspan is assigned, all affected cells in the span region are set to TRUE in the record table.
This way, only table settings in target rows need to be checked for rowspans.
Finally, since regions may span both columns and rows, both span types need to be checked in order to correct the 'target' column for regions to the right of the spanned region.
